The full lifecycle

From idea to production
in two skills.

No orchestration layer. No shared state. GitHub Issues, PRs, labels, and preview URLs are the only communication channel.

```
01

Scaffold or init

Starting fresh? factory new debates your tech stack, scaffolds the repo, sets up CI, and writes CLAUDE.md + FACTORY.md. Existing repo? factory init adds the config.

02

Talk to the PM

Describe what you want in plain language. The PM pushes back, researches your repo, and writes a spec you'd be proud to hand to any developer.

03

The factory builds

Plan ‚Üí implement ‚Üí open PR ‚Üí preview deploy. One step per cycle. Each cycle advances the state machine. Human checkpoints wherever you want them.

04

Preview and iterate

Every PR gets a live preview URL. The PM (or you) reviews the running app, leaves feedback. The factory fixes, preview updates. Tight loop until it's right.

05

Release and deploy

When you're ready, factory release tags the version and generates a changelog. factory deploy pushes to production with health checks and automatic rollback.

```
Configuration

FACTORY.md lives
in your repo.

Not a dotfile. Not hidden config. A visible, committed, versioned file that tells the factory how to build, preview, release, and deploy this repo.

Run factory init and it walks you through setup ‚Äî detects your language, hosting platform, merge settings, and existing conventions.

FACTORY.md 
## Workflow
- Branch naming: `factory/<issue>-<slug>`
- Merge strategy: squash
- Test command: pytest

## Human Checkpoints
- Auto-merge approved PRs: no
- Mark PRs ready for review: yes

## Previews
- Platform: vercel
- Comment URL on PR: yes

## Releases
- Versioning: semver
- Changelog: from merged PRs

## Production
- Deploy: `vercel --prod`
- Health check: /api/health
- Rollback: `vercel rollback`
Design principles

What we believe.

```

GitHub is the database

Issues are the work queue. PRs are the delivery mechanism. Labels are the state machine. Comments are the audit trail. No custom infrastructure needed.

One action per cycle

The factory does one thing then stops. This prevents runaway automation, keeps PRs small, and makes every action auditable. The next cycle picks up whatever comes next.

Never self-approve

The factory never approves its own PRs. Every change gets reviewed with fresh context ‚Äî either by a human or by a subsequent factory cycle that didn't write the code.

Config lives in the repo

FACTORY.md is committed, versioned, and visible. Anyone browsing the repo understands the automation. Rules grow from mistakes, not from upfront speculation.

```
